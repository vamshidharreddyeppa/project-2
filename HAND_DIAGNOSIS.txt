# Project 2 - Hand Diagnosis First

Source files used:
- sample-incident.log
- sample-dmesg.log
- sample-ps.txt

ROOT CAUSE:
The main issue was that the Ollama process loaded deepseek-7b FP16, which is the main reason why the model server was killed by the Linux OOM killer. Subsystem: MEMORY.

SINGLE OFFENDING LINE:
Jun 19 02:31:09 relief-gpu kernel: Out of memory: Killed process 8423 (ollama) total-vm:18402112kB, anon-rss:15994208kB, file-rss:0kB, shmem-rss:0kB, UID:1000 oom_score_adj:0

REASONING:
The most important clue is anon-rss:15994208kB on the OOM line. This means the process was holding about 15.3 GiB of memory. In a system where the RAM is around 16 GB, this much memory is more than 95% of the memory, so it is understandable why the model workload was killed. The earlier application logs also show that the service loaded deepseek-7b FP16, plus there was a warning that the KV cache was pre-allocated for context=32768 and batch=4. So the memory issue makes sense.

The process table also points to the same process. PID 8423 is ollama, and it is using 98.7% CPU with 15994208 KB RSS. The CPU usage shows that the model was working hard, but the actual reason it died was memory exhaustion. The kernel clearly invoked the OOM killer and killed ollama.

The storage messages are not the main problem. The log shows "nvme0n1: I/O timeout, retrying (attempt 1/3)" and then immediately says "nvme0n1: I/O retry succeeded". So that did not kill the service. The later systemd lines showing model-server.service failed and restarted are also after the OOM kill, so they are symptoms and not the first failure.

FIX:
Root-cause fix: As the currently loaded model is occupying too much memory, it is better to run a quantized model that fits the host. A practical first change would be to use a Q4_K_M model and lower the context from 32768 unless a long context is truly required.

Symptom patch only: restarting model-server.service. Restarting does not fix the sizing problem. It will load the same too-large model again, and the OOM killer can kill it again. So it will become a repeated process.

VERDICT:
This is not a random crash, an NVMe failure, or an apt update problem. It was a memory sizing failure. The simple reason is that this model is not suited for this machine because of the memory issue.
